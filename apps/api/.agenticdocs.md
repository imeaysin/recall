# 🏗️ API App Pragmatic Light-CQRS Architecture & Structural Guardrails

This document enforces the strict architectural blueprints and folder organization rules specific to this NestJS API application.

---

## 🧹 File Sanitization & Folder Normalization Rules

### 1. Eradicate DTO Layout Sprawl

- **The Defect:** Feature modules must never have an isolated `[feature].dto.ts` file sitting parallel to a `dto/` folder root.
- **The Action:** You must immediately merge all data contracts into a single, flat `dto/` directory at the module root and delete the stray file.

### 2. No Manual HTTP Error Leaking

- Domain services and calculators inside the `domain/` layer must **never** instantiate or throw raw NestJS HTTP Exceptions (like `BadRequestException`, `InternalServerErrorException`).
- **The Action:** They must throw pure, domain-specific semantic TypeScript exceptions that extend our base custom error framework. The centralized `HttpExceptionFilter` inside the global `common/` folder context will catch and map them to standard HTTP status codes at the API edge.

---

## 📂 Target Module Blueprint Layout

Every business module inside this application must match this exact flat architecture. No multi-level folder nesting is allowed:

src/[feature-name]/
├── domain/ # 🧠 Pure Business Layer (ZERO NestJS or ORM imports)
│ ├── [feature].model.ts # Strongly typed domain objects and models
│ └── [feature].calculations.ts # Pure mathematical transformations and state invariants
├── repository/ # 💾 Data Infrastructure (Light-CQRS Split)
│ ├── [feature].command.ts # Writes Path: Mutation queries, safe state changes, atomic transactions
│ └── [feature].query.ts # Reads Path: High-performance selects, reporting views, complex filters
├── dto/ # 🌐 Ingress Contracts (NO duplicate files outside)
│ ├── create-[feature].dto.ts
│ └── update-[feature].dto.ts
├── listeners/ # 🔀 Cross-Module Subscriptions (Decoupled Event Layer)
│ └── [feature].listener.ts # Event handlers using @nestjs/event-emitter
├── [feature].controller.ts # 🎛️ Ingress Routing, Guard Application, & Response Delivery
├── [feature].service.ts # ⚡ Orchestrator (Thin workflow runner coordinating Repos and Domain)
└── [feature].module.ts # 🔌 Module Wires and Dependency Providers

---

## 🔁 Cross-Module Communication via Event Listeners

- Feature modules must never cross-inject each other's services or repositories directly to avoid circular dependency runtime crashes.
- All cross-module workflows (e.g., deleting a User cascades into deleting all of that user's Notes) must use asynchronous event broadcasting:
  1. **Publisher:** Dispatches a typed event class using `EventEmitter2.emit()`.
  2. **Subscriber:** Catches the event using `@OnEvent('event.name', { async: true })` inside its local `listeners/` folder and passes execution safely to its own command repositories.
